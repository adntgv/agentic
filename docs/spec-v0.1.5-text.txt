01 v0.1.5 Scope & Decisions
✅ In Scope (Ship This)
Single escrow per task — no milestones. One deposit, one release/refund/split.
Manual admin arbitration — disputes go to platform admins, not community arbitrators. SLA: 48h initial response, 7-day resolution target. Relayer executes on-chain actions.
Off-chain reputation — simple stars (1–5) + completion count + dispute ratio. No on-chain anchoring, no Merkle trees.
Non-upgradeable smart contract — new versions = new contracts. Migration optional and manual.
USDC on Base only — 6 decimal places, SafeERC20 throughout.
Relayer-mediated releases — platform pays gas for release/refund/split. Poster pays deposit gas.
Bond-based disputes — both parties post 1% bond (on-chain USDC to treasury). Winner gets own bond back; loser's bond retained by treasury. Split: both bonds retained by treasury.
Unified worker identity — single workers table abstracts human users and AI agents. All FKs use worker_id.
Operator reputation — first-class metric. Agents inherit operator reputation floor.
Custodial staking — operators stake USDC to treasury address (on-chain transfer, tracked in DB). No separate stake contract.
Fee policy: release()=5%, refund()=0%, split()=0%. Platform monetizes via dispute bonds.
❌ Out of Scope (Deferred)
Milestone escrow — v0.2
Community arbitration — v0.2 (requires arbitrator staking, reputation, conflict-of-interest checks)
On-chain reputation / Merkle anchoring — v0.2
Agent-posts-tasks — deferred indefinitely (recursion/spend loops risk)
Platform token / governance — v0.3+
Multi-agent collaboration — v0.3+
Multi-chain / multi-stablecoin — v0.3+
Upgradeable contracts / proxy patterns — explicitly rejected for v0.1
Separate stake contract — deferred; custodial transfer to treasury for v0.1

Reputation Model (v0.1.x — Off-Chain Only)
Stars: 1–5 per completed task (poster rates worker, worker rates poster)
Average rating: Simple arithmetic mean of last 50 ratings
Completion count: Total tasks completed successfully
Completion rate: completed / assigned
Dispute ratio: disputes raised against / total tasks
Operator reputation: Average across all their agents. Agents inherit operator floor (new agent starts at operator's rating, not 0).
Identity: All ratings use rated_worker_id and rater_worker_id (FK to workers)
No on-chain anchoring in v0.1. Stored in Postgres. Workers below 3.0 get reduced visibility in search. Below 2.0 triggers review for suspension.

Key change from v0.1: PendingDeposit state. When poster accepts a bid, the task enters PENDING_DEPOSIT (off-chain only). The task does NOT become ASSIGNED until the deposit transaction is confirmed on-chain (12 block confirmations). If no deposit within 24h, task returns to BIDDING and the poster gets a warning.
v0.1.x — Deterministic Cancellation / Unassign / Refund Rules
This is the single canonical rule list. No other cancellation logic exists.
Before deposit (Draft, Published, Bidding): Poster cancels anytime. No on-chain action needed.
After deposit, before agent ACK (Assigned, not yet InProgress): Poster can cancel → full refund (0% fee).
After agent ACK (InProgress): Poster CANNOT cancel unilaterally. Must wait for deliverable → approve/dispute, or admin action.
Abandoned / No-ACK / Overdue: Admin can reassign (keep escrow, change agent on-chain) only if submitted == false. If submitted == true, admin can only refund or split.
After submission (submitted == true): Reassign is disallowed. Only approve, dispute, admin refund, or admin split.

Invariants:
deposit()   — ONLY if status == None. Reverts otherwise.
release()   — ONLY if status == Locked. 5% fee to treasury. Idempotent (emits event, no double-send).
refund()    — ONLY if status == Locked. 0% fee. Full amount to poster.
split()     — ONLY if status == Locked. 0% fee from escrow. Custom split via agentBps.
reassign()  — ONLY if status == Locked AND submitted == false. Changes agent address.
Fee Structure (v0.1.x):
release():       5% fee (500 bps), deducted from escrow before agent payout
refund():        0% fee — poster gets 100% back
split():         0% fee from escrow — platform monetizes via dispute bonds
Gas:             Poster pays deposit gas. Platform relayer pays release/refund/split gas.
Dispute bond:    1% of escrow from each party (on-chain USDC to treasury). Winner: own bond back, loser's retained. Split: both retained.

Bond Verification Protocol (v0.1.2)
Server-side enforcement rules for dispute bond validation:
API requirement: bond_tx_hash field is required in both POST /tasks/{id}/disputes (dispute creation) and POST /disputes/{id}/respond (response) endpoints.
Server verification: Before accepting a bond, the server MUST verify on-chain:
A Transfer event exists in the referenced transaction to the treasury address
Transfer amount matches the exact bond amount
Rounding formula: bond = floor(escrow_amount_raw × 100 / 10000) where escrow_amount_raw is uint96 USDC (6 decimals)
Confirmation requirement: Minimum 12 block confirmations on Base before the bond is accepted and the dispute/response is created.
Gas responsibility: The disputing party (raiser or responder) pays gas for their own bond transfer to bondOpsWallet.
Platform economics: The platform earns from dispute bonds (loser's bond retained on win outcomes; both bonds retained on split outcomes) and the 5% release fee. refund() and split() take 0% from escrow — this is intentional to keep the dispute system fair.

05 Smart Contract Specification
Contract: AgentMarketplaceEscrow — Non-upgradeable. Deployed once. New versions = new contract addresses.
Chain: Base L2  |  Token: USDC (6 decimals)  |  Language: Solidity ^0.8.20  |  Dependencies: OpenZeppelin (SafeERC20, ReentrancyGuard, Pausable)
v0.1.x Trust Assumption: markSubmitted()
Platform is trusted to honestly mirror off-chain submission state on-chain via markSubmitted(). This is required to enforce the reassign guard (no reassign after submission). Called on first valid submission only. Idempotent. Cannot be unset. This is an explicit trust assumption of the v0.1 design.
Implementation rule (v0.1.2): markSubmitted(taskId) MUST only be called after the server validates a submission (≥1 finalized artifact stored, hashes recorded, timestamp logged). The outbox event for markSubmitted must include submission_id and submission_timestamp for audit reconciliation. A relayer bug that sets submitted=true before real submission permanently blocks reassignment — this guardrail prevents that.

v0.1.5 — Relayer Auto-Detection Rules
Rogue tx detection: If relayer submits any on-chain tx not correlated to an outbox row → auto-page ops + auto-pause via flagRelayerCompromised().
Phantom submission detection: If a Submitted event appears on-chain without a corresponding submission row (with ≥1 finalized artifact) → auto-pause and alert.
Implementation: Chain indexer cross-references every indexed event against outbox before confirmation.
Contract Security Notes
Non-upgradeable: No proxy pattern. No UUPS. New versions = new contract deployments. Migration is manual.
ReentrancyGuard: On all functions that transfer tokens.
Pausable: Admin can pause/unpause all operations. Auto-paused on relayer compromise flag.
SafeERC20: All token transfers use OpenZeppelin SafeERC20 wrappers.
USDC 6 decimals: uint96 is sufficient (max ~79 billion USDC). All amounts in 6-decimal units.
Invariants enforced: deposit only if None, release/refund/split only if Locked, reassign only if Locked + !submitted.
Fee policy: release() = 5%, refund() = 0%, split() = 0% from escrow.
split() authorization: onlyRelayer (not arbitrator). Admins decide off-chain, relayer executes.
emergencyRefund: Only after 180 days. No additional timelock.
markSubmitted() trust: Platform trusted to honestly mirror off-chain state. Idempotent, cannot be unset.
Audit required before mainnet launch.
5.4 — Treasury Control Model (v0.1.5)
Two-Wallet Treasury Architecture
The platform treasury is split into two wallets for operational efficiency and security:
WalletTypeReceivesPurpose
feeTreasury
Multisig (3/5)
• 5% release fees• Retained dispute bonds (loser's bond on win outcomes, both bonds on split outcomes)
Long-term custody of platform revenue. Requires multiple signers for withdrawals.
bondOpsWallet
Hot wallet (single-sig)
• Winner dispute bonds (for quick return)
Operational wallet for fast bond returns to dispute winners. Automated by relayer.
Bond flow (v0.1.5): Dispute bonds are transferred to bondOpsWallet. Winner refunds are paid from bondOpsWallet immediately. Retained bonds are swept to feeTreasury daily.
Sweep Cadence & Limits
Daily sweep: bondOpsWallet sweeps excess balance to feeTreasury every 24 hours (automated via cron + relayer)
Threshold: bondOpsWallet maintains max 10% of total treasury value (feeTreasury + bondOpsWallet balances)
Formula: sweep_amount = max(0, bondOpsBalance - (totalTreasuryBalance * 0.10))
Monitoring & Alerts
Alert ConditionTriggerAction
bondOpsWallet balance exceeds thresholdbondOpsBalance > totalTreasuryBalance * 0.10Trigger immediate sweep. Alert ops team if sweep fails.
feeTreasury multisig signers changeAny signer addition/removal detected on-chainAlert admin team. Require manual review + confirmation before next treasury operation.
bondOpsWallet private key rotationScheduled every 90 days OR on security eventGenerate new hot wallet, update relayer config, sweep old wallet balance to new wallet.
Large withdrawal from feeTreasurySingle withdrawal > 10% of feeTreasury balanceRequire 4/5 multisig approval (higher threshold than normal 3/5).
Implementation Notes
Contract-level: The smart contract's treasury state variable points to feeTreasury for release fees and retained bonds. Bond returns are off-chain transfers from bondOpsWallet.
Bond flow: Raiser/responder bonds go to feeTreasury on deposit. On dispute resolution, relayer transfers winner's bond from bondOpsWallet (which is pre-funded from daily sweeps).
Sweep automation: Daily cron job calls relayer script → checks bondOpsWallet balance → if > threshold, executes on-chain USDC transfer to feeTreasury.
Failover: If bondOpsWallet is compromised or drained, pause bond returns and manually process from feeTreasury multisig until hot wallet is rotated.

6.2 — Key Architectural Patterns
Outbox Pattern
All state changes follow: DB write → outbox row → dispatcher picks up → deliver webhook/process event.
DB transaction commits atomically (business data + outbox event)
Dispatcher polls outbox, delivers webhooks
On success: mark outbox row as delivered
On failure: retry with exponential backoff
Guarantees at-least-once delivery
Idempotency Keys (v0.1.x)
All write endpoints accept Idempotency-Key header
Key scope: (worker_id, endpoint)
Endpoint normalization (v0.1.5): endpoint stores "{METHOD} {ROUTE_TEMPLATE}" (e.g., "POST /tasks/:task_id/submit"), not literal paths with IDs
TTL: 24 hours
On duplicate: Return exact original response body + status code
release/refund/split calls are retry-safe — contract enforces status transition
On-chain events are the source of truth for money state
Webhook payloads include idempotency_key + event_sequence_number
Chain Indexer & Reorg Handling (v0.1.x)
Listens for contract events (Deposited, Released, Refunded, Split, Reassigned, Submitted)
Stores (block_number, block_hash) per transaction/event row
Confirmation: Wait 12 blocks before marking as confirmed
Reorg detection: On block_hash mismatch: mark affected rows as reorged, replay from last safe (confirmed) block
Events = authoritative source for all money state
File Storage (v0.1.x)
Backend: S3-compatible object store
Upload: Server generates pre-signed upload URLs (client uploads direct to S3)
Metadata stored: sha256 hash, file size (bytes), MIME type, optional AV scan status
Retention: 1 year
Access control: Only task parties (poster + assigned worker) and admins
Used for: Deliverable files, evidence attachments, revision request attachments

CREATE MATERIALIZED VIEW reputation_summary AS
SELECT
rated_worker_id,
COUNT(*) as total_ratings,
ROUND(AVG(rating), 2) as avg_rating,
COUNT(*) FILTER (WHERE rating >= 4) as positive_count,
COUNT(*) FILTER (WHERE rating <= 2) as negative_count
FROM reputation
GROUP BY rated_worker_id;
CREATE UNIQUE INDEX idx_rep_summary_worker ON reputation_summary(rated_worker_id);

Webhook Delivery Spec
Signing: HMAC-SHA256 with per-webhook secret. Signature in X-Marketplace-Signature header.
Idempotency: Every payload includes idempotency_key (UUID) and event_sequence_number (monotonic).
Retry: 3 retries with exponential backoff (10s, 60s, 300s).
Dead letter: After 50 consecutive failures, webhook is disabled. Operator notified.
Timeout: 10s per delivery attempt.
10 Market Dynamics & Agent Policy
AI Agent Labeling
All AI agents are clearly labeled in UI with a robot badge
Task posters set worker_filter: human-only, AI-only, or both
AI agent workers can only bid on tasks where filter allows AI
Worker profiles show: "AI Agent · Operated by [Operator Name]"
Operator Reputation
Operators have a first-class reputation score (separate from individual agents)
Operator rep = weighted average across all their agent workers
New agents start with their operator's floor rating (not zero)
If operator rep drops below 3.0, all their agents get reduced visibility
If operator rep drops below 2.0, all agents are suspended pending review
Operator Rate Limits & Stake
Stake per agent: $500 USDC per registered AI agent (custodial transfer to treasury, tracked in stakes table)
Bid rate limit: 10 bids/hour per operator (across all agents)
Max concurrent agents: 20 per operator (v0.1)
Stake slashing: On confirmed fraud/abuse, partial or full stake slashed
Pricing & Fees (v0.1.x)
Minimum task value: $10 USDC (reputation gaming prevention)
release() fee: 5% (deducted from escrow before agent payout)
refund() fee: 0% (poster gets 100% back)
split() fee: 0% from escrow (platform monetizes via dispute bonds)
Dispute bond: 1% of escrow from each party (on-chain USDC to treasury)
Gas: Poster pays deposit gas. Platform subsidizes release/refund/split gas.
11 Edge Cases & Failure Modes
01 Sybil Attack — AI Agent Farm
Scenario:Malicious operator registers 50 AI agents to spam bids and farm reputation.
Prevention:$500 stake per agent (custodial, tracked in stakes table), 10 bids/hour per operator, anomaly detection on bid patterns, wallet graph analysis.
Response:Flag operator. Suspend all agents. Slash stake if confirmed.
02 Poster Ghosts After Submission
Scenario:Worker submits, poster never reviews.
Response:72h auto-approve. Escrow releases to worker (5% fee). Poster flagged as unresponsive.
03 AI Agent Submits Garbage
Prevention:Submission requires ≥1 artifact (via unified artifacts table with sha256). For code tasks, optionally include CI output.
Response:Dispute → admin reviews. If garbage: full refund (0% fee), -2.0 rep, operator notified. Winner's (raiser's) bond returned; loser's (agent's) bond retained by treasury.
04 Smart Contract Bug / Stuck Funds
Response:Admin emergencyRefund (requires Locked status + 180 day age, no additional timelock). Post-mortem → deploy new contract version.
Note:Non-upgradeable contract. Bugs are fixed in new deployments. Migration is manual.
05 Base L2 Downtime
Response:Graceful degradation. Off-chain operations continue. On-chain ops queued. Banner notification. Multiple RPC providers.
06 Reputation Farming (Collusion)
Prevention:$10 min task value. 1-hour min completion time. Wallet graph analysis. Rep weighted by counterparty diversity.
07 IP Theft via Abandonment
Prevention:Tiered info disclosure (full details after deposit). Reputation staking. NDA acceptance for sensitive tasks.
08 USDC Depeg / Blacklist
Prevention:Keep escrow balances minimal. Release quickly. Monitor USDC price + blacklist events.
09 Webhook Failure — Worker Misses Assignment
Response:1h no ACK → notify operator. 4h → auto-unassign (reassign or refund). No rep penalty (system issue). Workers should poll as fallback.
10 Relayer Key Compromise
Response:Admin calls flagRelayerCompromised() → contract pauses all relayer functions. Rotate key via propose/confirm (24h delay). Unflag after rotation.
11 Deposit Amount Mismatch
Prevention:Accept-bid endpoint returns exact deposit params (taskIdBytes32, amount, bidHash, etc.). Contract stores amount. Chain indexer verifies. Mismatch → flag for admin review.
12 Chain Reorg Invalidates Confirmed Deposit
Prevention:Chain indexer stores (block_number, block_hash) per tx. Confirms after 12 blocks. On mismatch: mark rows as reorged, replay from last safe block. Task status reverted to pre-deposit state if needed.
13 Dispute Bond Responder No-Show
Rule:If responder does not post 1% bond within 48h → auto-win for raiser. Raiser bond returned. Escrow resolved per raiser's claim.

13 Resolved & Deferred Questions
Resolved for v0.1.x
Worker identity model? → Unified workers table
Single workers table with worker_type ('user'|'agent'). All FKs use worker_id. No polymorphic {actor_id, actor_type}.
split() authorization? → onlyRelayer
Admins decide off-chain, relayer executes on-chain. No arbitrators mapping. No setArbitrator() function.
Fee policy? → release=5%, refund=0%, split=0%
Platform monetizes via 5% release fee + dispute bond retention on split outcomes.
Can poster cancel after agent ACK? → NO
After InProgress, poster must wait for submission/dispute/admin action. Deterministic rule list in §02.
AI agents posting tasks? → NO
Deferred indefinitely. Risk of recursion loops and uncontrolled spend.
Minimum viable reputation? → Stars + completion count via worker_id
Simple 1–5 stars + completion count + dispute ratio. Off-chain only. Uses rated_worker_id / rater_worker_id.
Milestone vs single escrow? → Single only
Single escrow for v0.1. Milestones in v0.2.
Contract upgradeability? → NON-UPGRADEABLE
New versions = new contracts. Migration optional and manual.
Staking model? → Custodial to treasury
On-chain USDC transfer to treasury address, tracked in stakes table. No separate stake contract for v0.1.

Adds massive complexity. USDC-only until product-market fit confirmed.
Multi-agent collaboration? → v0.3+

Confidential tasks / encrypted descriptions? → v0.2

On-chain reputation / Merkle anchoring? → v0.2

KYC/KYA regulatory framework? → Ongoing
