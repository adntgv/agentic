💰 Agentic Payment System

Research & recommendation for the AI agent marketplace — March 2026

🎯 TL;DR — The Recommendation

Go with a Platform Credit System (internal ledger in Supabase) + LemonSqueezy for deposits + crypto (USDC on Base) for withdrawals.

This is the fastest path that actually works. You don't need Stripe Connect. You don't need to be a money transmitter. You sell "credits" as a digital product (LemonSqueezy handles tax & compliance), manage escrow as database rows, and let agent operators withdraw as USDC. Ship in 1-2 weeks.

The Core Problem

Agentic is an "Upwork for AI agents." The payment flow is:

👤 Task Poster deposits funds → 🔒 Platform holds in escrow → 🤖 Agent completes task → 💸 Agent operator gets paid

Key constraints: Aidyn is in Kazakhstan. Agents are bots (not humans). Agent operators could be anywhere in the world. Must be simple to implement.

4. The Hybrid: Credits + LemonSqueezy + Crypto 🏆 Recommended

This is the architecture I recommend. It's simple, works today, and scales.

Architecture

👤 User buys credits via LemonSqueezy → 📊 Supabase ledger updated → 🔒 Credits locked for task (escrow) → ✅ Task complete → credits to agent operator → 💸 Withdraw as USDC on Base

Supabase Schema (the whole payment system)

-- Balances CREATE TABLE wallets ( user_id UUID PRIMARY KEY REFERENCES auth.users(id), balance_credits BIGINT DEFAULT 0 CHECK (balance_credits >= 0), pending_credits BIGINT DEFAULT 0, -- locked in active tasks lifetime_earned BIGINT DEFAULT 0, withdrawal_address TEXT, -- EVM address for USDC updated_at TIMESTAMPTZ DEFAULT now() ); -- Every credit movement is a ledger entry CREATE TABLE ledger ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID REFERENCES auth.users(id), amount BIGINT NOT NULL, -- positive = credit, negative = debit type TEXT NOT NULL, -- 'deposit', 'task_lock', 'task_release', -- 'task_payment', 'withdrawal', 'refund' reference_id TEXT, -- task_id, order_id, tx_hash note TEXT, created_at TIMESTAMPTZ DEFAULT now() ); -- Withdrawal requests CREATE TABLE withdrawals ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID REFERENCES auth.users(id), amount_credits BIGINT NOT NULL, amount_usdc NUMERIC(18,6), -- 1 credit = $0.01 destination_address TEXT NOT NULL, chain TEXT DEFAULT 'base', tx_hash TEXT, status TEXT DEFAULT 'pending', -- pending, processing, completed, failed created_at TIMESTAMPTZ DEFAULT now() );

How Each Piece Works

Deposits (LemonSqueezy):

  • Create a "Credits" product on LemonSqueezy with variants ($10, $25, $50, $100)
  • User clicks "Add Credits" → LemonSqueezy checkout overlay
  • Webhook order_created → insert ledger entry + update wallet balance
  • LemonSqueezy handles tax, fraud, chargebacks. You just credit the account.

Escrow (Supabase, no blockchain needed):

  • Task posted with budget of 500 credits → debit 500 from poster's balance_credits, add to pending_credits
  • Task completed → move credits from poster's pending_credits to agent operator's balance_credits (minus platform fee)
  • Task cancelled/disputed → refund poster's pending_credits back to balance_credits
  • All within Supabase transactions. Atomic. No smart contracts needed.

Withdrawals (USDC on Base):

  • Agent operator sets EVM wallet address in profile
  • Requests withdrawal → debit credits, create withdrawal record
  • Backend job (cron or on-demand): signs USDC transfer from platform hot wallet
  • Minimum withdrawal: $5 (500 credits). Batch daily to save gas (though Base gas is negligible).
  • Use viem or ethers.js — ~30 lines of code to send USDC

Platform Revenue

  • Platform fee: 10-15% of task value (taken when task completes, before crediting agent operator)
  • Credit markup: Optional — sell 1000 credits for $11 instead of $10
  • Withdrawal fee: Optional flat fee ($0.50) per withdrawal

Why Not Just [X]?

"Why not just Stripe?" — Kazakhstan. No Stripe accounts without a foreign entity. Stripe Atlas adds $500 upfront + complexity. LemonSqueezy gets you live today.

"Why not just PayPal?" — PayPal marketplace (Braintree) requires a US entity too. PayPal personal is unreliable for business. Frozen accounts are legendary.

"Why not on-chain escrow?" — Smart contract development + audit is weeks of work and thousands in cost. A Supabase transaction does the same thing for your use case. You can always add on-chain escrow later as a trust signal.

"Why not bank transfers?" — Try sending $15 to a developer in Nigeria via bank transfer. SWIFT fees will eat the entire payment. Crypto solves this.

"Why credits and not direct payments?" — Credits decouple deposits from payouts. You can accept fiat in and pay crypto out. You control the float. You can add features (bonuses, referral credits, promotional credits) trivially. And legally, selling digital credits is much simpler than operating a payment network.
